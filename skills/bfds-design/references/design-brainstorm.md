# 设计方向探索

设计方向探索只脑暴设计稿，不脑暴产品能力。它的产物是进入评审工作台的 A/B/C 三个三方向设计规格。

## 硬门禁

没有可信 `PRODUCT.md`、可信 `DESIGN.md`、以及已确认的目标界面与变更边界，不进入本阶段。

可以跳过“提问”，但不能跳过“脑暴”。如果材料足够，直接产出三个方向；如果材料不足，一次只问一个最高价值的设计表达问题。

## 输入检查

进入本阶段前必须能指向这些证据：

- `PRODUCT.md`：用户、产品目的、语气、设计原则、反参考。
- `DESIGN.md`：视觉系统、组件、token、布局、字体、动效边界。
- 用户材料：PRD、原型、截图、URL、Figma 或用户说明。
- 目标界面与变更边界确认：目标界面、现状来源、改动类型、必须保留、允许改变、必须避免。

任一缺失时回到对应门禁，不用设计方向探索补产品定义、后端方案或未确认的界面现状。

## 阶段检查清单

按顺序完成：

1. **上下文速读**：用 3-5 条列出本次设计必须服从的产品和视觉约束。
2. **不确定性扫描**：判断最大不确定性属于信息优先级、密度、状态表达、动效、设备优先级、品牌边界还是局部保留范围。
3. **提问或继续**：如果一个问题能显著降低设计风险，就问一个；否则直接进入三方向。
4. **生成设计方向**：产出 A/B/C 三个三方向设计规格。
5. **自检**：检查三个方向是否真的不同、是否遵守设计上下文和目标界面约束、是否没有越界产品能力。

## 提问边界

只问设计表达问题。不要把目标界面与变更边界门禁问题伪装成设计方向探索，也不要把产品能力问题带进来。

可问：

- 哪个信息应该最先被看见？
- 哪些内容要弱化、折叠或延后出现？
- 这个目标界面应该更高效、更安静、更强提示，还是更具品牌表现？
- 动效是低存在感反馈，还是承担引导/强调？
- 哪些状态必须高保真展示？
- 移动端、桌面端、嵌入场景哪个优先？
- 局部改动是否必须保留周边布局和节奏？

不可问：

- API 怎么设计。
- 数据模型怎么设计。
- 权限和角色怎么定义。
- 商业模式或产品范围是否扩大。
- 后端架构怎么拆。
- 目标界面到底是哪一个；这是目标界面与变更边界门禁，不是设计方向探索。

## 三方向设计规格

必须输出三个方向，字段齐全：

```json
{
  "optionId": "A",
  "name": "Quiet Precision",
  "designThesis": "用更安静的层级和明确状态减少输入焦虑。",
  "sourceConstraints": ["PRODUCT.md: 高频设置场景", "DESIGN.md: 保留现有 token"],
  "hierarchy": "主输入区第一优先，历史和说明弱化。",
  "density": "snug",
  "motion": "低存在感 focus 和验证反馈。",
  "stateTreatment": "error/loading/success 都在输入区附近明确呈现。",
  "layoutStrategy": "保留周边布局，只重排输入和帮助信息。",
  "interactionModel": "输入、保存、重置和错误解释在同一局部完成。",
  "visualSignature": "细分隔、稳定基线、单一强调动作。",
  "keep": ["页面导航", "提交逻辑"],
  "change": ["输入区层级", "辅助提示"],
  "avoid": ["新增无关设置项"],
  "risks": ["视觉记忆点偏少"],
  "bestFor": "高频设置和需要降低认知负担的产品 UI"
}
```

三个方向必须至少在两个维度上不同：

- 信息架构
- 层级和视觉重心
- 密度与节奏
- 状态表达
- 交互模型
- 动效承担的角色
- 局部保留范围

换色、换圆角、换阴影不算方向差异。

## 自检

生成工作台前，逐项检查：

- A/B/C 都引用了可信 `PRODUCT.md` / `DESIGN.md` 或目标界面约束。
- A/B/C 都没有新增未确认的产品能力。
- A/B/C 都写清必须保留、允许改变、必须避免。
- A/B/C 不是同一布局的换皮。
- 至少一个方向处理关键状态或关键交互。
- 如果跳过提问，说明“材料足够”的依据，而不是沉默跳步。

任一项不满足，先修正三方向设计规格，再进入评审工作台。

## 常见错误

| 错误 | 正确处理 |
|---|---|
| “材料足够，所以跳过设计方向探索” | 只能跳过提问，仍必须生成 A/B/C 三方向设计规格。 |
| 把目标界面与变更边界确认的问题当成设计方向探索 | 回到目标界面与变更边界门禁，先确认目标界面与变更边界。 |
| 三个方案只是换颜色 | 重新生成，至少两个设计维度必须不同。 |
| 用 brainstorm 补产品功能 | 停止，退回 PRD/OpenSpec/plan/superpower 等上游。 |
| 缺 `PRODUCT.md` / `DESIGN.md` 但继续出方向 | 停止，回到 设计上下文门禁。 |

## 收敛

把三个方向写入评审工作台页面说明。用户可以回复：

- “选 A”
- “选 B，但导航用 A 的”
- “C 的错误态加到 B”
- “都不对，重做方向”

用户选择后进入设计交付包生成。
